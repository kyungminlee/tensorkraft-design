This is exactly the kind of structural friction that separates standard tensor libraries from true quantum many-body frameworks.

The transition from Abelian to non-Abelian symmetries fundamentally breaks the assumption that the tensor product of two irreps yields a single target irrep.

In U(1) or $\mathbb{Z}_2$, the fusion rule is a simple scalar addition or XOR: $q_a + q_b = q_c$. It's a strictly $1:1$ (or $1:0$) mapping. In SU(2), the tensor product of two representations $j_1$ and $j_2$ decomposes into a direct sum of multiple representations according to the Clebsch-Gordan series:


$$j_1 \otimes j_2 = |j_1 - j_2| \oplus (|j_1 - j_2| + 1) \oplus \dots \oplus (j_1 + j_2)$$

If we try to force this into your current `tk-linalg` task generator, we hit a severe concurrency hazard. Let's break down the necessary architectural refactoring to handle this elegantly without destroying your Rayon threading model.

### 1. The Concurrency Hazard in `par_iter`

Currently, `SectorGemmTask` assumes an **input-centric** generation model: you iterate over the blocks in `A`, iterate over the blocks in `B`, find the single `out_key`, and dispatch the task.

If SU(2) fusion yields multiple `out_key`s, multiple different $(A_i, B_i)$ pairs will independently attempt to write to the same $C_k$ output block. If you dispatch this into Rayon's `par_iter()`, you will have a race condition. Resolving this with a `Mutex<DenseTensor>` around each output block would serialize execution and destroy your parallel speedup.

### 2. Architectural Shift: Output-Centric Task Generation

To maintain the lock-free, zero-contention parallel map, we must invert the task generation loop. Instead of asking, *"What does this pair of inputs produce?"*, we must ask, *"Which pairs of inputs contribute to this specific output sector?"*

This changes the task from a single GEMM to a **Composite GEMM Task** responsible for the complete lifecycle of one output block.

#### Redefining the Task Structures

```rust
/// Represents a single input pair that contributes to an output sector.
/// Includes the structural weight (e.g., from Wigner-Eckart / 6j symbols).
struct ContractionPair<'a, T: Scalar> {
    block_a: &'a DenseTensor<T>,
    block_b: &'a DenseTensor<T>,
    structural_weight: T::Real, // Evaluated Clebsch-Gordan/9j scalar
}

/// An output-centric task. Rayon guarantees this is the *only* thread 
/// writing to `out_key`, completely eliminating data races.
struct SectorCompositeTask<'a, T: Scalar> {
    out_key: SU2Irrep,
    /// All (A, B) pairs that fuse into this specific out_key
    contributing_pairs: Vec<ContractionPair<'a, T>>,
    /// Sum of all M*N*K FLOPs across all contributing pairs
    total_flops: usize,
}

```

### 3. Modifying the Generation Pipeline

The generation phase (Phase 1 of your block-sparse GEMM) needs to be split into a Map-Reduce builder pattern prior to Rayon dispatch:

1. **Map Possible Fusions:** Iterate over all $(A_i, B_j)$ blocks. Evaluate the SU(2) fusion rule to get the list of valid output irreps $[C_1, C_2, \dots, C_k]$ and their corresponding structural weights.
2. **Group by Output (The Builder):** Use a `HashMap<SU2Irrep, Vec<ContractionPair>>` to group these inputs by their target output sector.
3. **Construct Composite Tasks:** Convert the HashMap into a `Vec<SectorCompositeTask>`. Calculate `total_flops` by summing the $M \times N \times K$ cost of every pair in the `contributing_pairs` vector.

### 4. Preserving LPT Scheduling & Hardware Execution

Because we calculate `total_flops` at the composite level, your Longest Processing Time (LPT) scheduler remains perfectly valid. You simply sort the `Vec<SectorCompositeTask>` by `total_flops` in descending order before passing it to `into_par_iter()`.

The execution closure (Phase 2) now looks like this:

```rust
impl<'a, T: Scalar> SectorCompositeTask<'a, T> {
    fn execute<D: LinAlgBackend<T>>(&self, backend: &D) -> (SU2Irrep, DenseTensor<T>) {
        // 1. Allocate the output block C (initialized to zero)
        let mut block_c = allocate_zeroed_tensor(shape_c);

        // 2. Accumulate all contributing GEMMs into C
        for pair in &self.contributing_pairs {
            // C = (structural_weight) * A * B + 1.0 * C
            backend.gemm(
                T::from_real(pair.structural_weight), 
                &pair.block_a.as_ref(), 
                &pair.block_b.as_ref(), 
                T::one(), 
                &mut block_c.as_mut()
            );
        }

        (self.out_key, block_c)
    }
}

```

### 5. Integration with `tk-contract`

You correctly noted that `tk-contract` needs a `structural_contraction` callback.

With this output-centric architecture, the callback isn't just a passive observer; it acts as the routing engine during Step 1 of task generation. It takes the quantum numbers of $A_i$ and $B_j$, computes the Wigner-3j/6j/9j symbols (likely caching them using your `lie-groups` crate), and returns the iterator of `(SU2Irrep, Weight)` tuples that dictate which `SectorCompositeTask` receives the input pair.

**Summary of the Win:**
By shifting from "1 task = 1 GEMM" to "1 task = 1 Output Block," you completely sidestep the need for atomic accumulations or Mutexes, preserve the zero-copy GEMM architecture, keep the LPT scheduler perfectly intact, and isolate the SU(2) group theory logic purely into the task-generation phase.

This is exactly where the implementation complexity of non-Abelian symmetries typically peaks. Moving from U(1) to SU(2) doesn't just change how tensors are contracted; it fundamentally alters the physics of the entanglement spectrum and how we must handle basis truncation.

In your `tensorkraft` architecture, you've wisely isolated the non-Abelian extension into a `WignerEckartTensor` that separates the `ClebschGordanCache` (the structural/geometric part) from the `reduced` block-sparse tensor (the physical degrees of freedom).

Here is how the one-to-many fusion behavior completely reshapes the SVD truncation step, and the specific architectural changes required to handle it safely without corrupting the physics.

### 1. The Physics: Multiplet Degeneracy

In an Abelian system like U(1), a singular value $s_i$ corresponds to exactly one state in the Hilbert space. You gather all singular values across all $Q$-sectors, sort them globally, and slice the array at `max_bond_dim`.

In SU(2), the Wigner-Eckart theorem factors the tensor such that the actual SVD is performed **only** on the `reduced` blocks. However, a single singular value $s_{j, k}$ belonging to an irreducible representation (irrep) of spin $j$ actually represents a degenerate **multiplet** of $2j+1$ states in the full physical Hilbert space.

### 2. Architectural Impact 1: Multiplet-Weighted Sorting

If you just sort the raw singular values from the `reduced` blocks, your truncation error calculations will be completely wrong. Discarding a singular value of $0.05$ from a $j=0$ (singlet) sector discards $0.05^2$ of the state norm. Discarding the same $0.05$ from a $j=2$ (quintet) sector actually discards $5 \times 0.05^2$ of the norm.

**The Refactor:** The global sorting logic in `tk-dmrg` must become multiplet-aware. When collecting the singular values from all the `DenseTensor` blocks of the `reduced` tensor, you must tag each value with its irrep $j$ and compute its true physical weight:


$$w_{j,k} = (2j+1) \cdot s_{j,k}^2$$


The global sort for determining the truncation cutoff must be performed on these weighted values, not the raw $s_{j,k}$.

### 3. Architectural Impact 2: Multiplet Preservation (No Splitting)

In standard implementations, if `max_bond_dim` is 500, you keep exactly 500 states. In SU(2), you **cannot** split a multiplet. If your running total of retained states is 498, and the next singular value in the sorted list belongs to a $j=3$ (heptet, $2j+1=7$) multiplet, keeping it would push your bond dimension to 505.

If you truncate the multiplet and only keep 2 of the 7 states, you explicitly break SU(2) symmetry, and your DMRG sweep will likely collapse or converge to a non-physical state.

**The Refactor:** The `max_bond_dim` configuration must be treated as a "soft" target. The truncation algorithm must walk down the sorted list of singular values, accumulating the $2j+1$ dimensions. When the sum crosses `max_bond_dim`, the algorithm must make a binary choice: either keep the entire final multiplet (overshooting the target) or discard the entire multiplet (undershooting). This requires a strict loop invariant where truncation boundaries only ever occur between distinct multiplets.

### 4. Architectural Impact 3: `LinAlgBackend` API Friction

Currently, your `LinAlgBackend<T>` trait defines `svd_truncated` at the dense matrix level, taking `max_rank` directly.

This creates a serious architectural boundary issue for SU(2). The `DeviceFaer` or `DeviceCuda` backend implementations have no concept of quantum numbers or $2j+1$ multiplicities; they just see a flat array of `f64`. If you ask the dense backend to truncate to a specific rank internally, it cannot respect the multiplet boundaries.

**The Refactor (Two-Phase SVD):**
To keep the hardware-level numerics totally decoupled from the symmetry logic, the SVD process must be split:

1. **Phase 1 (Un-truncated SVD):** The `SparseLinAlgBackend` iterates over all valid $j$-sectors in the `reduced` tensor. It calls a non-truncating `svd` on the dense backend (or an SVD with a very generous noise-floor cutoff just to drop zeros).
2. **Phase 2 (Symmetry-Aware Global Truncation):** The `tk-dmrg` layer gathers all the `S` vectors, applies the $2j+1$ weights, sorts them, and finds the exact integer number of vectors to keep for *each specific $j$-sector* without splitting multiplets.
3. **Phase 3 (Slicing):** The `tk-dmrg` layer manually slices the $U$, $S$, and $V^\dagger$ matrices based on the exact multiplet-safe counts determined in Phase 2. The zero-copy `MatRef` views make this slicing operation practically free.

In heavily optimized numerical codebases, it is tempting to push the truncation logic as close to the hardware as possible to save allocation overhead. However, moving the truncation decision *up* the stack to the `tk-dmrg` layer ensures your dense BLAS micro-kernels stay perfectly generic and your symmetry guarantees remain mathematically rigorous.

The decision to push the Jordan-Wigner (JW) transformation into the Matrix Product Operator (MPO) construction phase, rather than embedding fermionic swap gates into the core contraction engine, is one of the most consequential performance optimizations in the `tensorkraft` architecture.

By doing this, the library trades a slight increase in one-time MPO compilation complexity for zero-overhead execution during the millions of tensor contractions that occur within the solver loop.

Here is a deep dive into how this is architected, why it works exceptionally well for 1D/Impurity models, and where the boundaries of this design lie.

### 1. The Physics: Non-Local Parity Strings

Fermionic operators anti-commute: $c_i c_j^\dagger + c_j^\dagger c_i = \delta_{ij}$. To map these to bosonic or spin operators (which commute on different sites and are what tensor networks naturally process), we use the Jordan-Wigner transformation. A creation operator at site $i$ is mapped to:


$$c_i^\dagger = S_i^+ \prod_{k<i} Z_k$$


where $Z_k$ is the parity operator (equivalent to the Pauli-Z matrix, $1 - 2n_k$) measuring whether a site has an even or odd number of fermions.

When a user defines a hopping term like $c_i^\dagger c_j$ with $i < j$, the physical operator evaluates to $S_i^+ Z_{i+1} Z_{i+2} \dots Z_{j-1} S_j^-$. The parity string $Z$ must span every site between $i$ and $j$.

### 2. Architectural Implementation: The MPO Automaton

In some frameworks, tensors are assigned "fermionic legs" carrying parity quantum numbers, and the contraction engine intercepts leg crossings to dynamically inject $-1$ signs. `tensorkraft` explicitly rejects this in favor of a purely bosonic contraction engine.

Instead, the JW string is permanently baked into the MPO matrices during the `OpSum -> MPO` compilation phase in `tk-dmrg`. This is typically done using a Finite State Automaton (FSA) or bipartite graph approach.

Here is how the architecture handles it:

* **AST Generation:** The `tk-dsl` macro parses the hopping term `Cdag_up(i) * C_up(j)` and generates a purely abstract Syntax Tree (AST) representing the operators.
* **Fermionic Detection:** When `tk-dmrg` compiles this AST, it reads the strong typing (`FermionOp::CdagUp`) and recognizes that the operators are fermionic.
* **State Machine Routing:** The MPO builder automaton transitions into a "JW String Active" state at site $i$.
* **Matrix Construction:** For every intermediate site $k$ ($i < k < j$), the MPO matrix $W^{[k]}$ is populated with the $Z$ operator in the specific matrix entry that connects the left state ($c_i^\dagger$ has happened) to the right state (waiting for $c_j$).
* **Termination:** At site $j$, the automaton places the $c_j$ operator and transitions back to the "Identity" ground state.

The resulting MPO tensors carry these sign factors as explicit matrix elements.

### 3. The Performance Win in `tk-contract`

By offloading the JW strings to the MPO, the `ContractionExecutor` and the `PathOptimizer` in `tk-contract` remain completely oblivious to the fact that they are simulating fermions.

* **Zero Branching Overhead:** The dense BLAS micro-kernels and the block-sparse GEMM routines do not need to check boolean parity flags or evaluate $(-1)^P$ on every block permutation.
* **Simplified Path Optimization:** The `CostMetric` in the `PathOptimizer` only has to evaluate memory bandwidth (strides) and FLOPs. It doesn't have to penalize paths that cause complicated fermionic leg crossings, because there are no fermionic legs.
* **Instruction Cache Preservation:** Keeping the contraction executor purely bosonic keeps the inner loop machine code compact, preventing instruction cache misses on the CPU.

### 4. Topological Limitations

The architecture document correctly identifies the fundamental limitation of this design.

* **1D and Star Geometries (Supported):** Because the Anderson Impurity Model maps a star geometry to a 1D chain via Lanczos tridiagonalization, the sites can be strictly linearly ordered. The JW string approach is correct and complete for these models.
* **2D PEPS and Trees (Unsupported):** In a 2D Projected Entangled Pair State (PEPS) or a tree tensor network, there is no single linear ordering of sites. Tensors will inevitably have legs that cross in ways that cannot be resolved by inserting a 1D string of $Z$ operators into an MPO.
* **The Mitigation:** The architecture wisely defers native fermionic swap gates to Phase 5+ behind a `fermionic-swap` feature flag. When activated, this will likely require a structural callback injected into the `tk-contract` engine, similar to how the SU(2) Wigner-Eckart symbols are evaluated.

The design of the `hamiltonian!{}` proc-macro is a masterclass in respecting the boundaries of the Rust compiler. In scientific computing, it is incredibly tempting to use metaprogramming to evaluate physics equations at compile time. However, the `tensorkraft` architecture explicitly forbids this, forcing the macro to act strictly as a syntactic boilerplate reducer.

Here is an engineering breakdown of how the macro pipeline generates the uncompressed operator sums (`OpSum`), why this two-phase design prevents massive build failures, and the architectural implications for the DSL.

### 1. The Trap: Compile-Time MPO Compression

Matrix Product Operator (MPO) construction requires substantial numerical linear algebra. To compress an arbitrary sum of local operators into a compact MPO, the system must build a finite-state automaton and perform repeated Singular Value Decompositions (SVD) to truncate redundant internal bonds.

If the `hamiltonian!{}` macro attempted to do this during the compilation of `tk-dsl`, it would trigger two catastrophic failures:

1. **Compiler Freezes:** Running iterative linear algebra inside `rustc`'s proc-macro execution context would stall compilation for minutes on complex lattices.
2. **Parameter Freezing:** Physics parameters like coupling constants ($J$, $U$, $V_k$) or system sizes ($N$) are usually read from runtime configuration files (e.g., JSON or passed via the Python bindings). A compile-time MPO builder would require hardcoding these parameters into the source text.

### 2. The Solution: The Two-Phase Pipeline

The architecture neatly sidesteps these issues by splitting the workflow across two crates with a strict dependency boundary.

**Phase 1: Compile-Time AST Generation (`tk-dsl`)**
The macro parses the domain-specific language (DSL) and expands it into standard Rust loop constructs and `OpSum` builder method calls. It performs *no* numerical computation.

For example, when the macro reads this:

```rust
sum i in 0..N-1 {
    J * (Sp(i) * Sm(i+1))
}

```

It emits pure Rust syntax-tree code roughly equivalent to:

```rust
let mut opsum = OpSum::new();
for i in 0..N-1 {
    opsum += J * op(SpinOp::SPlus, i) * op(SpinOp::SMinus, i+1);
}

```

**Phase 2: Runtime SVD Compression (`tk-dmrg`)**
Only at runtime, when `OpSum::compile_mpo(&self, backend, config)` is called, does the actual MPO compression occur. Because this happens in `tk-dmrg`, it has full access to the `LinAlgBackend` trait and the `SweepArena` allocators to perform the heavy SVD operations.

### 3. Type Safety via Strongly-Typed Operators

A classic source of runtime panics in string-based tensor libraries (like standard ITensor) is a typo in an operator name, such as `op("S_plu", i)`. This error is only caught when the program reaches that specific line of execution.

`tensorkraft` eliminates this by mapping the DSL tokens directly to strongly-typed enums (`SpinOp::SPlus`, `FermionOp::CdagUp`). If a user types an invalid operator in the macro, `rustc` catches it immediately during AST expansion. Furthermore, the `CustomOp<T>` escape hatch allows users to inject arbitrary dense matrices for non-standard models without breaking the type system.

### 4. Lattice Routing and the Snake Path

The macro also parses the lattice geometry (e.g., `lattice: Chain(N = 100, d = 2)`).

DMRG is inherently a 1D algorithm. To simulate 2D systems like a `Square` or `Triangular` lattice, the geometry must be flattened. The `Lattice` trait achieves this by providing a `dmrg_ordering()` snake-path mapping. The `hamiltonian!{}` macro uses this trait to seamlessly translate 2D coordinates $(x, y)$ into the 1D index $i$ expected by the `OpSum` builder.

### Actionable Architectural Feedback

* **Macro Variable Hygiene:** Proc-macros must resolve variables defined outside their scope (like $J$ or the array $V[k]$ in the Anderson model example). You must ensure the macro expansion uses proper span hygiene so it doesn't accidentally shadow local variables or fail to capture the physics parameters correctly.
* **Error Messaging:** When a user inevitably writes invalid DSL syntax (e.g., forgetting a `+` between terms), proc-macros often output cryptic, unhelpful `rustc` errors pointing to the macro invocation rather than the specific line. Implementing custom `syn::Error` reporting within the `hamiltonian!{}` parser is critical to point the compiler's red squiggly line exactly at the user's typo.

The integration of Longest Processing Time (LPT) scheduling into the block-sparse BLAS dispatch is one of the most practical engineering decisions in the `tensorkraft` document. It directly addresses a hardware-utilization pathology that silently cripples many naive parallel tensor network implementations.

Here is an architectural breakdown of why this block-sparse dispatch strategy is necessary, how it operates, and the physical constraints it overcomes.

### 1. The Physics: The Binomial Starvation Problem

In models with Abelian symmetries like U(1) particle number conservation, the Hilbert space is partitioned into distinct symmetry sectors. Because of the combinatorial nature of quantum states, these sector sizes follow a binomial distribution: there are a few massive blocks near the center (e.g., half-filling or $S_z = 0$) and many tiny blocks trailing off at the edges.

If you dispatch a block-sparse tensor contraction to a parallel thread pool in standard storage order (typically sorted sequentially by quantum number), the scheduler will eventually hand a massive central block to a single thread while the other threads quickly burn through the tiny edge blocks. The result is "long-tail starvation": most of your CPU cores idle while waiting for a single thread to finish a massive matrix multiplication.

### 2. The Architectural Solution: Three-Phase LPT Dispatch

To resolve this, the architecture completely decouples the mathematical execution order from the memory storage order using a classic LPT heuristic.

The `SparseLinAlgBackend::block_gemm` implementation separates the work into three distinct phases:

* **Phase 1: Task Generation:** The system iterates over all combinations of input sectors from tensor A and tensor B. If the quantum numbers validly fuse, it calculates the estimated FLOP cost ($M \times N \times K$) and packages the operation into a `SectorGemmTask`. No computation happens here.
* **Phase 2: LPT Scheduling & Execution:** The vector of tasks is sorted in descending order based on FLOPs. Rayon's `into_par_iter()` then consumes this sorted list. The heaviest tasks are dispatched first, saturating all cores immediately. As threads finish these massive GEMMs, Rayon's work-stealing scheduler automatically feeds them the lighter tasks from the tail end of the list, ensuring perfect core saturation until the very end.
* **Phase 3: Structural Restoration:** Because the output blocks are generated out-of-order, they violate the strict `PackedSectorKey` sorting invariant required for the $O(\log N)$ binary searches. The final step simply re-sorts the collected output tuples back into contiguous structural alignment.

### 3. Engineering Strengths

* **Zero Interior Mutability:** By relying on `into_par_iter().map().collect()`, the architecture completely avoids `Mutex`, `RwLock`, or `UnsafeCell` blocks during the parallel accumulation.
* **Rayon Synergy:** LPT is the optimal heuristic to pair with a work-stealing scheduler. Work-stealing excels at cleaning up many small, irregular tasks, which perfectly matches the tail of the LPT queue.

### 4. Actionable Feedback & Architectural Risks

* **Peak Memory Spikes:** By scheduling the absolute largest GEMM tasks simultaneously at the very beginning of Phase 2, you force the `SweepArena` to allocate the largest possible output blocks $C_k$ concurrently across all threads. If memory is tight (especially in GPU pinned memory), this simultaneous peak could exhaust the arena capacity, whereas a sequential execution would have reused space. You should ensure the arena sizing accounts for $N_{cores} \times \text{MaxBlockSize}$.
* **Cache Thrashing on the Memory Bus:** If 16 or 32 physical cores all simultaneously initiate massive dense matrix multiplications, they will heavily contend for L3 cache bandwidth. While the FLOP utilization will look perfect on a profiler, the actual throughput might be memory-bound. You may need to tune the `ThreadingRegime` thresholds to fall back to a single heavily-threaded BLAS call (Regime 1) slightly earlier than expected to better utilize shared cache.
* **Task Generation Overhead:** The double loop checking `compute_fusion_rule(*key_a, *key_b)` runs in $O(N_a \times N_b)$ time. For systems with thousands of small sectors (e.g., 2D models mapped to 1D), this sequential setup phase might become a noticeable overhead before the parallel work even begins.

The interaction between Longest Processing Time (LPT) scheduling and the `ThreadingRegime` logic elegantly solves one of the most notorious performance traps in numerical computing: thread pool oversubscription.

When mixing a work-stealing task scheduler (like Rayon) with a multithreaded BLAS backend (like `faer` or OpenBLAS), careless design leads to a nested parallelism disaster. If Rayon spawns $N$ threads to process independent tensor blocks, and each of those threads calls a BLAS routine that attempts to spawn another $N$ threads, the system suddenly has $N^2$ active threads fighting for CPU time. This causes catastrophic context-switching overhead and L3 cache thrashing, tanking performance.

Here is an architectural breakdown of how `tensorkraft` prevents this by dynamically coupling the LPT queue to the hardware dispatch strategy.

### 1. The Dynamic Dispatch Switch (`ThreadingRegime`)

Rather than relying on a static thread configuration, the `ThreadingRegime::select` method evaluates the structural metadata of the `BlockSparseTensor` against the physical core count of the machine at runtime. It routes the execution into one of two distinct physical regimes:

* **Regime 1: Fat Sectors (Dense-Dominated):** Triggered when the maximum sector dimension exceeds 500 and the total number of sectors is less than the number of CPU cores. In this regime, Rayon is entirely disabled. The LPT-sorted tasks are executed sequentially (`into_iter`), and the BLAS backend is given the full machine thread pool to parallelize the heavy internal matrix multiplications.
* **Regime 2: Fragmented Sectors (Sparsity-Dominated):** Triggered when there are many small sectors. The BLAS threading is forced to 1, and Rayon takes over via `into_par_iter` to distribute the independent sector GEMMs across all available cores.

### 2. Execution Pipeline Interaction

The LPT logic and the `ThreadingRegime` do not operate in isolation; they form a strict sequential pipeline:

1. **Task Generation:** A `Vec<SectorGemmTask>` is populated with all valid $A_i \times B_j$ sector combinations and their estimated FLOP counts.
2. **Unconditional LPT Sort:** The tasks are sorted in descending order by FLOPs. This happens regardless of the threading regime.
3. **Regime Divergence:**
* If **Fragmented Sectors** is chosen, the LPT sorting is what prevents Rayon's long-tail starvation. The heaviest block GEMMs are pushed into the Rayon work queues first, keeping all cores saturated, while the tiny blocks at the end of the array are used by the work-stealing scheduler to fill idle cycles as threads finish their heavy tasks.
* If **Fat Sectors** is chosen, the LPT sort simply ensures that the sequential execution tackles the largest matrix multiplications first. While this doesn't impact CPU saturation (since BLAS owns the threads), it optimizes the memory allocation pattern in the `SweepArena`, requesting the largest memory blocks early before fragmentation can occur.


4. **Structural Restoration:** The outputs are re-sorted by their `PackedSectorKey` to restore the binary-search invariants.

### Actionable Architectural Feedback

* **Brittle Heuristics:** The threshold `max_sector_dim > 500` is hardcoded. A $500 \times 500$ matrix multiplication takes milliseconds on a modern CPU. On a workstation, $500$ might be a good threshold to switch to BLAS threading, but on a massive HPC node (e.g., dual-socket 128-core architectures), giving 256 threads to a single $500 \times 500$ GEMM will result in severe thread synchronization overhead that outweighs the compute. This threshold must be dynamically calibrated against the L3 cache size per core and the total core count.
* **The "Middle Ground" Pathology:** The logic creates a sharp performance cliff. Consider a scenario on a 16-core machine with a tensor containing 17 sectors, where 4 of them are massive ($D \approx 800$) and the rest are tiny. Because `tensor.n_sectors() < n_cores` evaluates to false ($17 \not< 16$), the system routes to **Fragmented Sectors**. Rayon gets activated, and BLAS is forced to 1 thread. You will now process four $800 \times 800$ GEMMs using exactly one thread each, completely abandoning the highly optimized SIMD multi-threading of the underlying BLAS library.
* **Mitigation Strategy:** The `ThreadingRegime` should not be a binary switch based on the *entire* tensor's shape. Instead, the task vector should be partitioned. The top $K$ heaviest tasks (the "head" of the LPT list) should be dispatched sequentially to multithreaded BLAS, and only the "tail" of small tasks should be handed to single-threaded Rayon.

Integrating asynchronous GPU execution into a CPU-bound, work-stealing thread pool is notoriously difficult. If a CPU thread submits a kernel to the GPU and then blocks (waits) for the result, that physical CPU core is effectively dead to the rest of the application. If all Rayon threads block on GPU tasks, the CPU starves, and overall throughput collapses.

The `tensorkraft` document touches on a stream-aware `ContractionExecutor` and a hybrid CPU/GPU routing strategy. To safely fuse this with the Longest Processing Time (LPT) Rayon scheduler without starving the CPU, the architecture must evolve into a **Heterogeneous Asynchronous Dispatcher**.

Here is the architectural map of how to adapt your thread pool to safely feed `DeviceCuda`.

### 1. Arithmetic Intensity Routing (The CPU/GPU Split)

GPUs excel at massive throughput but suffer from high kernel launch latency (typically 2–5 microseconds). A $10 \times 10$ block-sparse GEMM will finish on a CPU cache before the GPU driver even finishes parsing the launch command. The architecture correctly notes the need for a dimension threshold to route operations.

This must be integrated directly into the Task Generation phase:

* **Task Profiling:** During Phase 1, the task generator computes the FLOP cost ($M \times N \times K$) of each `SectorGemmTask`.
* **The Router:** If the cost exceeds a calibrated `gpu_flop_threshold`, the task is tagged for `DeviceCuda`. If it is below the threshold, it is tagged for `DeviceFaer`.
* **Split Queues:** The single LPT queue is split into two queues: a `GPU_Queue` (sorted by LPT) and a `CPU_Queue` (sorted by LPT).

### 2. The GPU Submission Thread (Isolating the Driver)

Do not let Rayon worker threads touch the CUDA runtime API. The NVIDIA driver maintains its own hidden state and locks; having 16 Rayon threads simultaneously hammering `cublasDgemm` or `cudaStreamCreate` will cause lock contention inside the proprietary driver, stalling your worker threads.

* **Dedicated Submitter:** Spawn a single, dedicated GPU submission thread. Its only job is to pop tasks from the `GPU_Queue` and issue asynchronous launches to the CUDA driver.
* **Stream Pooling:** The submission thread manages a pool of CUDA streams (e.g., 4 or 8 streams). It maps independent tensor contractions to different streams, allowing the GPU's hardware scheduler to overlap GEMMs and maximize Streaming Multiprocessor (SM) occupancy.
* **Fire and Forget:** Because cuBLAS calls are asynchronous, the submission thread never blocks. It issues the GEMM, records a `cuda::Event`, and immediately moves to the next task in the `GPU_Queue`.

### 3. Asynchronous Dependency Resolution

The critical hazard occurs when a CPU task in `tk-contract` depends on the output of a GPU task.

If `ContractionNode B` (CPU) needs the output of `ContractionNode A` (GPU), the CPU cannot proceed until the data is transferred back to the `SweepArena`.

* **Avoid Spin-Waiting:** A Rayon thread cannot call `cudaEventSynchronize(event_A)` because that blocks the OS thread.
* **Continuation Passing:** The DAG executor must become asynchronous. When the GPU submission thread launches `Node A`, it attaches a callback or uses a lightweight polling mechanism.
* **Data Transfer (DMA):** Once the `cuda::Event` signals completion, an asynchronous memory copy (`cudaMemcpyAsync`) brings the data into the pinned `SweepArena` buffer.
* **Rayon Unlocking:** Only *after* the DMA transfer event is signaled does the system push `Node B` into the `CPU_Queue` for Rayon to pick up.

### Actionable Architectural Refinements

* **Batched cuBLAS for the "Middling" Tail:** The document mentions using batched cuBLAS for block-sparse sector GEMMs. This is crucial. Instead of sending 50 medium-sized tasks to the `CPU_Queue`, the router should group them by identical $(M, N, K)$ dimensions and dispatch them as a single `cublasDgemmBatched` call. This circumvents kernel launch overhead and reclaims GPU dominance for the middle of the LPT curve.
* **Pre-Pinned Arenas:** To make the async DMA transfers work without stalling the CPU, the destination buffers *must* be in page-locked memory. Your `PinnedMemoryTracker` and `SweepArena` fallback to pageable memory if the budget is exhausted. If a task is routed to the GPU but its `SweepArena` buffer fell back to pageable memory, the CUDA driver will force a synchronous, blocking copy under the hood, instantly destroying your asynchronous overlap. You must add a hard constraint: if pinned memory is exhausted, all subsequent tasks must be dynamically re-routed to the `CPU_Queue`.

===

The decision to replace the Gaussian window with an exponential window $\exp(-\eta|t|)$ in the v6.0 revision is mathematically and physically rigorous.

Linear prediction fundamentally assumes the signal is a sum of decaying complex exponentials. For gapped/insulating phases, the Green's function naturally decays, and linear prediction excels. However, for metallic phases (Fermi liquids), the Green's function $G(t)$ exhibits power-law decay (e.g., $t^{-1}$), which severely violates the linear prediction ansatz and causes the Toeplitz pseudo-inverse to become pathologically ill-conditioned.

Here is an architectural and mathematical evaluation of how `tensorkraft` handles this instability, along with critical blind spots in the deconvolution phase.

### 1. The Physics: Artificial Lifetime Broadening

**Strength:** From a physics perspective, multiplying $G(t)$ by $\exp(-\eta|t|)$ is not just a numerical hack; it maps perfectly to introducing an artificial scattering rate (or inverse lifetime) $\eta$ to the quasi-particles. This forces the signal into the exponentially decaying basis that linear prediction expects.

Furthermore, correcting the Fourier transform to a Lorentzian $2\eta/(\eta^2+\omega^2)$ makes the entire pipeline mathematically consistent.

### 2. Algorithmic Evaluation: The Deconvolution Trap

The architecture specifies that the induced Lorentzian broadening is deconvolved post-FFT via a simple division in frequency space:


$$A_{\text{true}}(\omega) \approx A_{\text{windowed}}(\omega) \cdot \frac{\eta^2+\omega^2}{2\eta}$$

**Actionable Risk 1: Catastrophic High-Frequency Amplification**
The deconvolution multiplier $\frac{\eta^2+\omega^2}{2\eta}$ is a parabola that grows quadratically with $\omega$. At low frequencies (near the Fermi level, $\omega \approx 0$), this is perfectly stable. However, at high frequencies (the tails of the spectral function), the true spectral weight is nearly zero, consisting mostly of numerical noise from the SVD truncation and the finite $\Delta t$ Trotter/TDVP error. Multiplying this noise by $\omega^2$ will violently amplify it, creating massive unphysical peaks at the edges of the spectrum.

* **Mitigation:** You cannot use raw parabolic deconvolution. You must implement a high-frequency cutoff or a tapered deconvolution filter (like a Wiener filter) that smoothly turns off the deconvolution factor when $\omega$ exceeds the physical bandwidth of the lattice.

**Actionable Risk 2: Positivity Violations**
A physical spectral function (density of states) must be strictly non-negative: $A(\omega) \ge 0$. Direct frequency-space deconvolution of noisy signals frequently produces negative oscillations (ringing artifacts). If $A_{\text{true}}(\omega)$ dips below zero, it will crash the DMFT self-consistency loop during the bath discretization step, as you cannot map a negative density of states to an Anderson Impurity Model.

* **Mitigation:** You must enforce a positivity constraint post-deconvolution. Simply clamping negative values to zero ($A = \max(A, 0)$) breaks the spectral sum rules ($\int A(\omega) d\omega = 1$). You will need a rescaling step to restore the sum rule after clamping, or ideally, formulate the linear prediction directly with a positivity constraint.

### 3. Hardware / Performance Implications

**Actionable Risk 3: Brute-Force Toeplitz Inversion**
The architecture specifies "SVD-regularized pseudo-inversion" with a noise floor cutoff for the prediction matrix. The prediction matrix in linear prediction is a Toeplitz matrix (constant diagonals).

* If your `LinAlgBackend` just passes this matrix to `DeviceFaer`'s `gesdd` (the standard divide-and-conquer SVD), you are performing an $O(N^3)$ operation where $N$ is the number of time steps.
* **Mitigation:** Toeplitz systems can be inverted in $O(N^2)$ using the Levinson-Durbin recursion, or even $O(N \log^2 N)$ with superfast algorithms. While SVD regularization is necessary for rank-deficient noise filtering, relying purely on dense SVD will make the linear prediction step a severe CPU bottleneck if the maximum simulation time (`max_time`) requires tens of thousands of $\Delta t$ steps.

### 4. Cross-Validation Safety Net

**Strength:** The absolute best architectural decision in this section is the mandatory Chebyshev cross-validation. Because Chebyshev expansion computes the spectral function directly in the frequency domain, it entirely bypasses the time-domain linear prediction instabilities. Using this as a ground-truth check ensures that if the $\eta$ deconvolution distorts the physics, the solver catches it rather than silently polluting the DMFT loop.
