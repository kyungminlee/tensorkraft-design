# Response to v6.0 External Review (Seventh Cycle)

## Overview

Six points raised: three accepted with concrete document changes, one accepted as a debug-mode guard, two acknowledged for future phases with documentation added. No structural redesign required.

---

## 1. Deconvolution Noise Amplification — Accepted. Concrete fix applied.

The naive deconvolution formula A_true(ω) = A_windowed(ω) · (η²+ω²)/(2η) grows as ω² at high frequencies. At ω = 100η, the amplification factor is ~5000× — any numerical noise in the spectral tails becomes catastrophic.

The v6 document said "subject to a noise-floor cutoff" but never specified the mechanism. v7 adds two concrete regularization strategies:

1. **Hard frequency cutoff** (`deconv_omega_max`): beyond ω_max, the deconvolution factor is clamped to 1.0. Default: 10× the impurity model bandwidth.

2. **Tikhonov-style damping** (`deconv_tikhonov_delta`): the denominator is modified from 2η to (2η + δ·ω²). This bounds the amplification factor to 1/δ regardless of frequency, smoothly tapering the correction at high ω instead of hard-cutting.

Both parameters are added to `LinearPredictionConfig`. The Chebyshev cross-validation remains the ultimate check.

## 2. SVD `gesdd` Silent Inaccuracy — Accepted as debug-mode guard.

`gesdd` can return without signaling an error while producing inaccurate small singular values for pathologically ill-conditioned matrices. A `debug_assert!` on the reconstruction residual ‖A − UΣV†‖/‖A‖ < 1e-10 catches this in debug/test builds. Compiled out in release (`--release`), adding zero production overhead.

In practice, the Tikhonov regularization already handles the downstream consequence (bad small singular values get suppressed by δ). The `debug_assert!` ensures we detect the problem during development rather than silently relying on the regularization to mask it.

## 3. MPI Mode B Load Imbalance — Acknowledged. Documentation added.

The `MPI_Allgather` convergence barrier causes fast ranks to idle while slow ranks finish. For the Phase 4 target (single-orbital Bethe lattice), impurity solver iteration counts are relatively uniform — the variance is modest. For multi-orbital or cluster DMFT with heterogeneous bath parameters (Phase 5+), mitigation strategies are documented in §10.3: asynchronous `MPI_Iallgather`, dynamic work-stealing, and adaptive mixing.

## 4. Pinned-Memory Fallback Telemetry — Accepted. Small change, high value.

The `log::warn!` is upgraded to a structured telemetry event with a global `PINNED_FALLBACK_COUNT` atomic counter. The counter is exposed in `DMRGEngine` stats so monitoring dashboards and Jupyter notebooks can surface the 2× bandwidth regression. A silent performance cliff that wastes days of debugging is now an immediately visible metric.

## 5. NUMA-Blind Pinned Memory — Acknowledged. Documented for Phase 5+.

On multi-socket nodes, `cudaMallocHost` may allocate on a NUMA node remote from the GPU's PCIe root complex, throttling DMA bandwidth by 30–50%. For single-GPU (Phase 5 target), the OS typically handles this correctly. For multi-GPU (Phase 5+ NCCL extension), §10.2.6 documents the requirement to bind allocations to the correct NUMA node via PCIe bus ID → NUMA node mapping.

## 6. SU(2) Fusion-Rule One-to-Many — Accepted. Sharpest observation in the review.

The Abelian `compute_fusion_rule(*key_a, *key_b) -> Option<PackedSectorKey>` returns at most one sector per input pair. For SU(2), j₁ ⊗ j₂ = |j₁−j₂| ⊕ ... ⊕ (j₁+j₂) yields *multiple* output sectors. The `SectorGemmTask` generation loop (§5.3.1) must fan out to `Vec<SectorGemmTask>` per input pair, with each task weighted by Clebsch-Gordan coefficients.

This is documented in §4.4 as a known refactoring requirement scoped to the `su2-symmetry` feature flag. The `structural_contraction` callback handles coefficient evaluation, but the task-generation fan-out is a structural change to the LPT scheduling phase. The Abelian code path is unaffected.
