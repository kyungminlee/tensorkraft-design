# Response to v3.0 External Review

## Overview

Both critique points are accepted and integrated into v4.0. One is a clean concurrency bug (monitor thread leak). The other is a scaling violation that, while slightly overstated in the review, leads to the correct conclusion (matrix-free projection is mandatory). Below is my detailed reasoning.

---

## Critique 1: Signal-Monitor Thread Leak

**Verdict: Accepted in full. This is an unambiguous concurrency bug.**

The v3 monitor thread has no exit condition on the happy path. If the solver runs for five hours and completes normally, the monitor thread continues looping forever — waking every 100ms, re-acquiring the GIL, and calling `check_signals()` on a solver that no longer exists. When the Python interpreter begins teardown, the dangling thread calls `Python::with_gil` on a destroyed interpreter state, producing a segfault.

The `mpsc::channel` with `recv_timeout` is the idiomatic Rust fix. The channel serves as both a polling timer and a shutdown signal in a single construct:

- `recv_timeout(100ms)` returns `Err(Timeout)` → poll for Ctrl+C.
- `recv_timeout` returns `Ok(())` → solver finished, exit cleanly.
- `recv_timeout` returns `Err(Disconnected)` → `done_tx` was dropped (solver panicked), exit cleanly.

The `done_tx.send(())` after `py.allow_threads` returns guarantees termination on the happy path. When `done_tx` is dropped during a panic, `Disconnected` guarantees termination on the unhappy path. Panic safety is automatic.

**One addition beyond the reviewer's fix:** I added `monitor_handle.join()` after sending the done signal. Without this, there's a narrow race window: the monitor thread might still be inside `Python::with_gil` when the next `solve()` call begins, or when the Python interpreter starts teardown at script exit. The `join()` blocks until the monitor thread has fully exited, closing this window. The cost is negligible — the monitor thread exits within one `recv_timeout` cycle (≤100ms) after receiving the done signal.

## Critique 2: Subspace Expansion O(D⁶) Trap

**Verdict: Core point accepted. Scaling analysis slightly overstated but conclusion is correct.**

The reviewer correctly identifies that explicitly constructing the null-space projector P_null = (I − A_L · A_L†) is wasteful. The matrix-free sequential projection is strictly superior. However, the stated O(d²D⁶) scaling requires clarification.

### Corrected scaling analysis

A_L at a center site with bond dimensions D_left ≈ D has shape (dD × D). The operations in question:

| Operation | Explicit projector | Matrix-free |
|:----------|:-------------------|:------------|
| A_L · A_L† | O(dD · D · dD) = O(d²D³) | — |
| P_null · \|R⟩ | O((dD)²) = O(d²D²) | — |
| O = A_L† · \|R⟩ | — | O(dD · D) = O(dD²) |
| \|R_null⟩ = \|R⟩ − A_L · O | — | O(dD · D) = O(dD²) |
| **Total** | **O(d²D³)** | **O(dD²)** |

The explicit projector path costs O(d²D³), not O(d²D⁶). The O(D⁶) scenario would require forming *and inverting* the full (dD × dD) projector, which nobody would do — the projector is idempotent and used multiplicatively, not inverted.

**However, the reviewer's conclusion is completely correct.** The matrix-free path at O(dD²) is a factor of d·D cheaper than the explicit path at O(d²D³). For D = 1000, d = 4, that's ~4000×. Inside the innermost TDVP loop — which executes once per bond per time step, potentially millions of times during a DMFT real-time evolution — this is the difference between "viable" and "unusable."

The two-step recipe is now the mandated implementation in §8.1.1:

1. Compute overlap: O = A_L† · |R⟩ (one mat-vec, O(dD²))
2. Subtract projection: |R_null⟩ = |R⟩ − A_L · O (one mat-vec, O(dD²))

The explicit projector is explicitly forbidden in the code comments and documented as a scaling trap in the architecture text.

### Testing

A new test case in §12.1 verifies equivalence between the matrix-free and explicit approaches on small test cases (where the explicit form is tractable), and checks orthogonality ⟨A_L|R_null⟩ ≈ 0 to confirm the projection is correct.
