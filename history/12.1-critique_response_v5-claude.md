# Response to v4.0 External Review (Fourth Cycle)

## Overview

All six critique points are accepted. Three are Rust-specific engineering issues (arena lifetimes, monomorphization, string typing). Three are physics-level refinements (fermionic sign rules, subspace expansion oscillation, linear prediction instability). None require structural redesign — they are additions, clarifications, and parameter extensions to existing abstractions.

---

## Critique 1: Arena Lifetime Trap — Accepted. Documented, not redesigned.

The reviewer identifies the Phase 1 boss fight: arena-allocated intermediates carry `&'a` lifetimes tied to the `SweepArena`, but the final SVD output must outlive the arena reset and live in the permanent `MPS` struct.

The fix is already latent in the `TensorCow` design — `Borrowed` vs `Owned` is exactly the temporary/persistent distinction — but the v4 document never made the ownership transfer explicit. v5 adds §3.3.2 with `into_owned()` semantics and marks step 4 of the §9 data flow table as the ownership boundary.

Critically, this is a problem the borrow checker *solves for free*. You literally cannot compile code that holds a borrowed arena reference past `SweepArena::reset()`. The pain is in satisfying the checker, not in debugging a runtime error. This is Rust working as intended.

## Critique 2: Compile-Time Explosion — Accepted with layered strategy.

The monomorphization concern is real but the recommendation to use `dyn LinAlgBackend` at the sweep level needs care. Making `LinAlgBackend` fully object-safe requires type-erasing the `Scalar` parameter, which defeats the purpose. The v5 approach is layered:

- Inner loops (GEMM, matvec): static dispatch, non-negotiable.
- Sweep scheduler: static by default, `dyn`-eligible if needed.
- `tk-python`: enum dispatch over a small set of feature-gated combinations.
- CI: `cargo-llvm-lines` monitoring with a 60-second per-crate threshold.

The pragmatic reality is that most users need exactly `f64 + U1 + DeviceFaer`. The problem is theoretical until Phase 3 when the full DMRG engine is compiled. Profile then, restructure only if necessary.

## Critique 3: String-Typed Operators — Accepted. Clear improvement.

`op("S+", i)` is a footgun. `op("S_plus", i)` would silently fail or panic during MPO compilation. Typed enums (`SpinOp::SPlus`, `FermionOp::CdagUp`) catch typos at compile time and cost nothing. The `CustomOp` escape hatch via `Into<SiteOperator<T>>` preserves flexibility for non-standard models. This goes into `tk-dsl` from day one.

## Critique 4: Fermionic Sign Rule — Accepted as documentation.

The contraction engine is bosonic-only. Jordan-Wigner strings are encoded in the MPO. This is correct and complete for all 1D geometries through Phase 4. The v4 document left this implicit; v5 makes it explicit in §6.4 with a clear statement of the limitation and the deferred `fermionic-swap` feature flag for tree/PEPS.

## Critique 5: Bond-Dimension Oscillation — Accepted. Subtle and important.

If injected null-space vectors have singular values near the truncation threshold, hard D_max truncation discards them immediately, and the next expansion re-injects them — producing an oscillating, discontinuous bond dimension. The soft D_max policy allows temporary overshoot (default 1.1×) with exponential decay over `dmax_decay_steps` (default 5), giving new entanglement time to either grow into significant singular values or decay naturally. Two new parameters in `TdvpStabilizationConfig`.

## Critique 6: Linear Prediction Gaussian Windowing — Accepted. Standard practice.

For metallic G(t) ~ t⁻¹, the Toeplitz prediction matrix is poorly conditioned regardless of SVD regularization. Gaussian windowing W(t) = exp(−ηt²) enforces decay, stabilizing the pseudo-inverse. The Lorentzian broadening in frequency space is deconvolved post-FFT. Chebyshev cross-validation confirms the deconvolution is physically correct. New `gaussian_broadening_eta` parameter in `LinearPredictionConfig`, defaulting to 0.0 (disabled for gapped phases).
