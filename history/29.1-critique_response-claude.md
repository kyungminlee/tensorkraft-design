**1. Flat-Buffer Reallocation During TDVP Subspace Expansion — Accept**

This is an excellent catch and represents a genuine design conflict that the document doesn't address. The flat-buffer refactor (Phase 4) and TDVP subspace expansion (also Phase 4) are on a collision course: subspace expansion mutates tensor structure (appending columns to A_L, growing sector blocks), and a flat buffer makes structural mutation O(D_total²) instead of O(D_sector²) because every downstream offset must shift.

The reviewer's proposed solution — distinguish "mutation layout" from "compute layout" and flatten only after expansion is complete — is exactly right. During the TDVP step, tensors live in the current `Vec<DenseTensor<T>>` fragmented layout where structural mutation is cheap. Before shipping to GPU or entering the GEMM phase, a `flatten()` method packs the blocks into a contiguous buffer. This is a one-time O(D_total²) copy per contraction, which is negligible relative to the O(D³) GEMM it feeds.

The `FlatBlockStorage` as currently specified should be understood as the *compute-side read-only representation*, not the canonical mutable storage. The document needs to make this dual-layout strategy explicit.

**2. SVD debug_assert Silenced in Release — Accept with modification**

The reviewer is right that a multi-week DMFT run with zero SVD validation is risky. Silent corruption of small singular values by `gesdd` is rare but real, and the Tikhonov regularization will stabilize the numerics while the physics drifts.

However, I'd push back slightly on the proposed fix. A randomized trace estimator is clever but adds implementation complexity. A simpler approach: check the residual every K-th SVD call (e.g., K=1000) in release builds, controlled by a runtime config flag. At K=1000, the overhead is 0.1% of SVD calls, and since the residual check is O(mn·min(m,n)) — the same complexity as the SVD itself — the amortized cost is negligible. This is simpler to implement than a trace estimator and catches the failure mode within ~1000 sweep steps of it occurring.

**3. GIL Polling via `ctrlc` Crate — Reject**

This sounds clean but is wrong for our use case. The `ctrlc` crate (and raw `SIGINT` handlers in general) have a fundamental problem: they operate at the *process* level, not the Python interpreter level. When tensorkraft is loaded as a Python extension via PyO3, the Python interpreter has already installed its own `SIGINT` handler. Installing a competing handler via `ctrlc` would either: (a) replace Python's handler, breaking `Ctrl+C` behavior for the rest of the Python process, or (b) chain with Python's handler, introducing undefined behavior around handler ordering.

The current design — a dedicated monitor thread polling `py.check_signals()` every 100ms — is the PyO3-recommended pattern. The reviewer's concern about "lock contention disrupting other Python threads" is overstated: `Python::with_gil` in CPython is a simple mutex acquire/release that takes ~100 nanoseconds. Doing this once every 100ms is a duty cycle of 0.0001%. The GIL is already the bottleneck for Python multithreading; an additional 100ns acquisition every 100ms is unmeasurable against the background of normal GIL contention from Python's own interpreter loop (which acquires/releases the GIL every ~5ms for thread switching).

**4. SU(2) Task Generation Memory Spike — Partially accept**

The concern about combinatorial explosion of `Vec<SectorGemmTask>` for SU(2) is theoretically valid but practically overstated for the target physics. In SU(2)-symmetric DMRG, the number of irreps at a given bond is bounded by D_reduced (the number of distinct j values), which is typically 10–50 even at large bond dimensions. The fan-out per input pair is at most 2j_max + 1 output sectors, so the total task count is O(D_reduced² × j_max). For j_max = 10 and D_reduced = 50, that's ~250,000 tasks at ~64 bytes each = ~16 MB. This is well within L3 cache and poses no allocation pressure.

That said, documenting this bound and adding a capacity pre-allocation (`Vec::with_capacity`) based on the estimated fan-out is good practice. A lazy iterator pipeline would add complexity without measurable benefit for realistic parameters. I'll add a note on pre-allocation and the bound analysis.

**5. Soft D_max Time vs Iteration Ambiguity — Accept**

This is a subtle and important point. The current `expansion_age` field counts discrete time steps (incremented by `tick_expansion_age`), but the TDVP integrator may use adaptive time stepping where `dt` varies. If `dmax_decay_steps = 5.0` means "5 iterations" and the integrator shrinks dt by 10× near a sharp feature, the bond dimension decays 10× faster in physical time — coupling a numerical tolerance to a physical relaxation rate. The decay should be in physical time units, tracked by accumulating `Σdt` rather than counting iterations.
